- [x] Settings screen (`SettingsView`, mockup frame 10), opened from Home's profile button. - [ ] Light mode pass across all screens (dev spec §7) — partially done, see below. **Found and fixed a real bug**, not just a verification gap: `RepSetForgeTheme` had a separate `LightColors` enum that was never referenced anywhere in the app (confirmed via grep — zero call sites) — every view hardcoded `RepSetForgeTheme.Colors.X`, the *dark* palette, unconditionally. Picking "Light" in Settings only flipped `.preferredColorScheme` (system chrome) while every custom-drawn surface/text color stayed dark — light mode was non-functional. Fixed by merging `LightColors` into `Colors` as dynamic `Color`s backed by `UIColor(dynamicProvider:)`, resolving per `UITraitCollection.userInterfaceStyle` — every one of the ~20 existing call sites (`RepSetForgeTheme.Colors.X`) now correctly repaints with zero call-site changes needed. Also swept the whole app for hardcoded hex/`.white`/`.black` bypassing the theme (`grep -rn "0x[0-9A-Fa-f]\{6\}"` outside `RepSetForgeTheme.swift`, plus `Color.white`/`Color.black`/literal `.foregroundStyle(.black)` etc.) — the only hits are intentional fixed-black text on the signal-green button background (mockup convention, verified via the WCAG formula below: 12.67:1 dark-mode signal, 6.93:1 light-mode signal, both comfortably pass), nothing else bypasses the theme.
  **Contrast measured, NOT fixed** (computed via the WCAG relative-luminance formula against the mockup's own light-mode hex values, since there's no simulator in this environment to eyeball it): `textPrimary`/`textSecondary` against `surface`/`surfaceRaised`/`surfaceInput` all clear ≥5.1:1 — fine. But `textTertiary` (0x98A0AC), `signal` (0x1FA968), `pr` (0xB8860B), `warn` (0xFF7A59), and `destructive` (0xFF5D5D) all fall short of 4.5:1 against both light surfaces (measured 2.4–3.3:1 depending on the pair — `pr`/`signal`/`destructive` clear the 3:1 *large-bold-text* threshold against `surfaceRaised` but not the 4.5:1 normal-text one; `textTertiary` and `warn` fail even that). These are the mockup's own light-mode CSS custom-property values (not something introduced by the SwiftUI translation), so this isn't a translation bug to silently correct — but mechanically darkening them to hit 4.5:1 turned out to distort the palette badly (`warn` drifts to a saturated red barely distinguishable from `destructive`, breaking the CLAUDE.md invariant "Warning orange... red only for destructive actions"), so that's not a safe unilateral fix either. Left as an explicit, data-backed open item for whoever has visual sign-off (real `gymchalk-tokens.json`, or a designer/simulator pass) rather than guessing at new brand hex values blind.
