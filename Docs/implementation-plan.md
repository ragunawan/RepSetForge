# RepSetForge Implementation Plan

## Current Architecture Summary

The repository was intentionally emptied before this implementation. The new app is a fresh SwiftUI iOS 17+ application with:

- SwiftUI for UI and navigation.
- SwiftData for local persistence. CloudKit private sync is documented as a production follow-up because the fresh schema currently runs in local-only mode for reliable simulator validation.
- Protocol-backed services for Health export, Live Activity coordination, rest timing, session persistence, PR calculation, import/export, and recommendations.
- A centralized native design system derived from `docs/repsetforge-hifi.html`.
- Unit test target generated by `generate_project.py`.

## Data And Service Requirements

- `Exercise`: user-created exercise catalog, canonical dedup key, muscle/equipment metadata.
- `Routine` / `RoutineItem` / `ProgressionRule`: routine library, double progression, 5/3/1, percentage waves, and RIR autoregulation prescriptions.
- `WorkoutSession` / `SessionExercise` / `SetEntry`: active and completed workout logging.
- `PRRecord`: derived personal records.
- `BodyMetric`: bodyweight and body-fat series.
- `AppSettings`: unit, rest, RPE, plates, theme, Health, iCloud status.
- Services: session autosave/restore, recommendations, PRs, progression prescriptions, Watch telemetry contracts, rest timer, CSV import/export, HealthKit export, ActivityKit Live Activity, notifications.

## Dependencies Between Tasks

- Screens depend on models/services and design tokens.
- Active workout depends on session persistence, rest timer, PR engine, and settings.
- Health export and Live Activity depend on session commit and rest state.
- Progress/History depend on completed session aggregation.
- Routine update prompt depends on summary diffing.

## Risks And Ambiguities

- A real CloudKit private database, HealthKit write validation, Live Activities, and notifications require signed entitlements/device testing. The code includes production-facing interfaces and simulator-safe fallbacks.
- The v1.1 Watch companion now has a buildable watchOS target and phone-side contracts. Real HKWorkoutSession mirroring, WatchConnectivity payload delivery, Smart Stack/AOD polish, and reconnection/backfill validation still require signed Apple Watch testing.
- The v1.1 progression methodologies are named but not fully parameterized. This pass implements conservative defaults: canonical 5/3/1 wave prescriptions from training max, configurable percentage waves, and RIR load adjustment from the last completed set.
- The wireframe includes dense web-style miniature charts. Native charts are implemented with lightweight SwiftUI shapes to avoid adding dependencies.
- Superset auto-scroll and full drag-reorder depth are partially adapted to native controls in this pass.
